Components

Structure & responsibility

Single responsibility

One component, one job

Doing too much

function UserCard() {
  const [user, setUser] = useState(null)

  useEffect(() => {
    fetch('/api/user')
      .then(r => r.json())
      .then(setUser)
  }, [])

  // fetches + renders + formats
  // + handles error + loading...
  return <div>...</div>
}

Split concerns

// hook: just fetches
function useUser(id) {
  return useQuery({
    queryKey: ['user', id],
    queryFn: () => getUser(id),
  })
}

// component: just renders
function UserCard({ user }) {
  return <div>{user.name}</div>
}
If describing this component takes the word "and", it's doing too much.

Co-location

Keep things together that change together

components/
  UserCard/
    UserCard.tsx          ← component
    UserCard.test.tsx     ← tests here, not in __tests__/
    UserCard.module.css   ← styles here, not in styles/
    index.ts              ← re-export for clean imports
  • Delete the component, everything goes with it
  • No hunting across folders to find related files
  • A test next to the component is a reminder to write it
Move files that change together. Separate files that change independently.

Composition over config

Small and flexible beats big and opinionated

Mega-props

<Button
  icon="star"
  iconPosition="left"
  loading={true}
  badge={3}
  badgeVariant="error"
  size="lg"
  variant="primary"
/>

Composable

<Button>
  <StarIcon />
  Save
  <Badge variant="error">3</Badge>
</Button>
  • Consumers control structure, no guessing which prop to use
  • New combinations don't require new props on the base component

Don't over-abstract

The rule of three

Extract only after writing it three times.

  • First time: just write it
  • Second time: tolerate the duplication
  • Third time: you understand the pattern now, extract it
Before abstracting: do these cases actually belong together, or am I forcing a pattern?
"Duplication is far cheaper than the wrong abstraction." — Sandi Metz

State Management

Where state lives

State proximity

Start local. Go global only when you must.

// Step 1: local state (most state lives here)
const [isOpen, setIsOpen] = useState(false)

// Step 2: lift up when two siblings need the same state
// (shared parent owns it, passes down as props)

// Step 3: Context when many components need the same state
// (theme, locale, auth user)

// Step 4: Zustand / Jotai for complex global state
// (rare, and earned)
Most state never needs to leave the component. The farther it travels, the harder it is to reason about.

Server state vs UI state

Different problems. Different tools.

UI State Server State
isOpen, activeTabuser data, product list
isDarkModecart items, notifications
form inputsearch results
useState / ContextReact Query / SWR
Don't store server data in useState. You'll re-invent caching, loading, and stale-data handling badly.

Avoid prop drilling

...but don't reach for global state too fast

Drilling through 3+ levels

<App>
  <Dashboard user={user}>
    <Sidebar user={user}>
      <Avatar user={user} />
    </Sidebar>
  </Dashboard>
</App>

Options

// 1. Composition: pass components
<Dashboard>
  <Sidebar>
    <Avatar /> {/* fetches own data */}
  </Sidebar>
</Dashboard>

// 2. Context (when truly shared)
const user = useUser()

// 3. Component fetches its own data

Server Communication

Fetching data, done right

Separate fetching from rendering

Mixed together

function UserProfile() {
  const [user, setUser] = useState(null)

  useEffect(() => {
    fetch('/api/user/1')
      .then(r => r.json())
      .then(setUser)
  }, [])

  return <div>{user?.name}</div>
}

Separated

function useUser(id) {
  return useQuery({
    queryKey: ['user', id],
    queryFn: () => getUser(id),
  })
}

function UserProfile({ id }) {
  const { data } = useUser(id)
  return <div>{data?.name}</div>
}
  • Hooks are testable in isolation
  • React Query gives caching, deduplication, and refetching for free

Loading & error states are first-class

Every fetch has 3 states. All 3 are UX.

function UserProfile({ id }) {
  const { data, isLoading, error } = useUser(id)

  if (isLoading) return <Skeleton />
  if (error) return <ErrorMessage error={error} />

  return <Profile user={data} />
}
If you don't handle loading and error, the feature is not done.

CSS

Tokens, utilities, and responsiveness

Utility-first vs semantic CSS

Two valid approaches. Pick one. Be consistent.

Tailwind (utility-first)

<button className="
  px-4 py-2 rounded
  bg-blue-500 text-white
  hover:bg-blue-600
  disabled:opacity-50
">
  Save
</button>

CSS Modules (semantic)

<button className={styles.primaryButton}>
  Save
</button>

/* UserCard.module.css */
.primaryButton {
  padding: 0.5rem 1rem;
  background: var(--color-brand);
  border-radius: var(--radius-md);
}
TailwindCSS Modules
Fast to write, no namingClean JSX, explicit naming
Verbose className stringsMore context-switching

Design tokens

One source of truth. Never hardcode values.

Magic values everywhere

.card  { border-radius: 8px; padding: 16px; }
.modal { border-radius: 8px; padding: 24px; }
.btn   { border-radius: 8px; color: #1a1a2e; }

Tokens

:root {
  --radius-md: 8px;
  --color-text: #1a1a2e;
  --space-md: 16px;
  --space-lg: 24px;
}

.card  { border-radius: var(--radius-md); padding: var(--space-md); }
.modal { border-radius: var(--radius-md); padding: var(--space-lg); }
.btn   { border-radius: var(--radius-md); color: var(--color-text); }
Change the token, change it everywhere. Consistency becomes automatic.

Responsive by default

Mobile-first. Add complexity as screen grows.

.grid {
  display: grid;
  grid-template-columns: 1fr;        /* mobile */
}

@media (min-width: 768px) {
  .grid {
    grid-template-columns: 1fr 1fr;  /* tablet */
  }
}

@media (min-width: 1024px) {
  .grid {
    grid-template-columns: repeat(3, 1fr); /* desktop */
  }
}
Start with the smallest screen. Override upward. Desktop-first leads to max-width soup.

Cross-cutting concerns

TypeScript & folder structure

TypeScript at boundaries

Type the edges. Let inference do the rest.

// Type API responses (you don't control these)
type User = {
  id: string
  name: string
  role: 'admin' | 'user'
}

// Type component props (your public interface)
type UserCardProps = {
  user: User
  onFollow: (id: string) => void
}

// Don't over-type internals, let TypeScript infer
const doubled = [1, 2, 3].map(n => n * 2)  // number[] ✓
// const doubled: number[] = ...             // redundant
Types are documentation the compiler checks. Focus them on API responses, props, and function signatures.

Folder structure

Feature-based scales. Type-based breaks.

Type-based (breaks at scale)

src/
  components/
    # 200 files
    # no way to know what
    # relates to what
  hooks/
    # another 80 files
  utils/
    # who knows

Feature-based (scales)

src/
  features/
    auth/
      components/
      hooks/
      api/
    dashboard/
      components/
      hooks/
      api/
  shared/
    components/   ← truly reusable
    hooks/
    utils/
If you delete a feature, how many folders do you touch? Ideally one.

The golden rule

Write code for the next developer.
  • Would you understand this without comments?
  • Does the component name tell you what it does?
  • Is the folder structure obvious to someone who didn't write it?
  • Would a deletion leave no orphaned files behind?
