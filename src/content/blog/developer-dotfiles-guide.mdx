I've kept a [dotfiles repo](https://github.com/phoinixi/dotfiles) for years, but it changed shape recently. Until last year it was the usual stuff: brew, zsh, a Brewfile, some macOS defaults. Now it provisions two things side by side: my human shell and the Claude Code agent harness I use every day. Both come up from a single `./install.sh` on a fresh machine.

This post is the short tour.

## What's in it now

The repo is split into modules. The dispatcher (`install.sh`) runs them in order, and you can pick a single module if you only want to refresh part of the setup:

```bash
./install.sh                  # everything
./install.sh --only claude    # re-symlink the agent harness
./install.sh --only zed
./install.sh --only ghostty
```

Layout, trimmed to the parts that matter:

```
Brewfile                brew bundle source
.zshrc                  shell config
install.sh              dispatcher with --only <module>
install/                numbered modules (00-brew, ..., 60-macos)
git/                    gitconfig + global gitignore
osx/                    macOS defaults
claude/                 Claude Code harness
zed/                    editor config
ghostty/                terminal config
mcp/                    MCP server templates
bin/scaffold-ai         per-project bootstrap CLI
AGENTS.md               contract for agents modifying this repo
```

## The human side

Pretty standard for a 2026 frontend setup, but a few things changed since the last version of this post:

- **Terminal**: [Ghostty](https://ghostty.org) instead of Hyper. Faster, native, no Electron.
- **Editor**: [Zed](https://zed.dev) as the primary, VS Code kept around as backup. Zed's agent panel and the Claude Code CLI cover most of what I used Cursor for.
- **CLI tools**: `eza`, `bat`, `ripgrep`, `fzf`, `git-delta`, `zoxide`, `atuin`, `gh`, `httpie`, `jq`/`yq`. `atuin` and `zoxide` are the two I'd miss most if I lost them.
- **Node**: `fnm` for version management, `pnpm` and `npm` as package managers.

The Brewfile declares all of this. Re-running `brew bundle` from the install script is idempotent, so nothing reinstalls if it's already there.

## The agent harness

This is the part that didn't exist a year ago. The `claude/` directory in the repo gets symlinked to `~/.claude/`, which is where Claude Code reads its config from:

- `settings.json` - sanitized, durable preferences only
- `CLAUDE.md` - global agent memory, loaded into every session
- `agents/`, `commands/`, `hooks/`, `output-styles/` - my authored cognitive tools
- `plugins/installed_plugins.json` - declares enabled plugins (`caveman`, `vercel`, `coderabbit`)

The trick is the sanitization. Claude Code's live `settings.json` mixes two things: durable preferences (what plugins are on, what edit mode I want) and machine-specific bits like absolute-path permission allowlists. Only the durable part is tracked. The local part lives in `~/.claude/settings.local.json`, which is gitignored, and the install script never touches it.

Before committing pulled changes, I run `./scripts/sync-claude-settings.sh` and it diffs live vs tracked, telling me exactly what would get stripped. Stops me from accidentally checking in someone else's project paths.

## Per-project scaffolding

`bin/scaffold-ai` is a small CLI the install script puts on `$PATH`. In any new repo:

```bash
mkdir my-app && cd my-app
scaffold-ai .
```

It drops in `.claude/settings.json`, `.claude/commands/`, `.mcp.json`, `AGENTS.md`, `.editorconfig`, and `.gitignore`, with `{{PROJECT}}` replaced by the directory name. Every new project is agent-ready in about three seconds.

## Secrets

Stored in macOS Keychain. The `.zshrc` exposes a one-line helper:

```sh
kc OPENAI_API_KEY        # prints the secret to stdout
$(kc GITHUB_TOKEN)       # use inline as a value
```

Add a new one with `security add-generic-password -a "$USER" -s NAME -w`. No `.env` files anywhere, no `1Password` CLI dependency, swap to Bitwarden or doppler later without touching the rest of the repo.

## Why bother

The honest answer: I'd been treating Claude Code's config as throwaway, redoing it on every machine. Once I had three machines and started syncing slash commands and hooks between them by hand, it was time. Treating the agent harness as a peer to the shell, in the same repo, with the same install path, removed a whole category of "wait, where did I configure that" moments.

If you're starting from scratch, fork it or just steal the parts you like.
